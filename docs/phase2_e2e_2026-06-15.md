# Phase 2 — E2E-Smoke + Worker-Recovery (2026-06-15)

> Session-Protokoll (Seminarfach-Material). Bezug: `docs/PLAN.md` §5 Phase 2.
> Alles read-only gegenüber IServ — keine Buchung, kein Submit (PLAN §6).

## 1. End-to-End-Smoke (Modus A), headless auf dem VPS

Skript: `automation/e2e_smoke.py` (`uv run python -m automation.e2e_smoke`,
Server muss laufen). Treibt die echten Seiten `host.html` + `scan.html`
per Playwright durch den kompletten Flow.

**Ergebnis: bestanden.** Verifizierte Kette:

1. Host-Login → `main-view`, WebSocket verbunden.
2. Klassenliste geladen (24 Klassen aus der read-only ausleihe-api).
3. Klasse gewählt → Schüler-Queue aufgebaut (30 Schüler, alphabetisch).
4. Helfer angelegt → QR-Code (PNG-Data-URL) + Scan-URL mit Token.
5. Scanner-Seite (separater Browser-Context = anderes Gerät) → WS verbunden,
   Helfer erscheint im Host als „verbunden".
6. „Nächster" → Playwright-Worker loggt ein, lädt die Schülerkartei
   (read-only), Scanner zeigt Schüler inkl. Bezahlt-Status und Bücherzahl.
7. Schüler im Host „Aktiv".
8. Simulierter Scan über WS → Worker füllt das Barcode-Feld → Status `staged`
   (**kein Enter, keine Buchung**, PLAN §6).

### Gefundener Bug (gefixt)

`web/scan.html`: Das Schüler-Panel blieb unsichtbar, obwohl die Daten ankamen.
Ursache: `studentPanel.style.display = ''` entfernt nur das Inline-Style; die
CSS-Regel `#student-panel { display:none }` greift dann wieder. Fix: explizit
`= 'block'`. Re-Test grün.

### Bekannte, harmlose Meldung

Beim Laden von `host.html` (vor Login) erzeugt der `/api/state`-Probe eine
`403`-Konsolenmeldung — das ist die Login-Erkennung der Seite, kein Fehler.
Browserseitig nicht unterdrückbar ohne Extra-Endpoint; bewusst belassen.

## 2. Worker-Recovery (Re-Login bei Session-Ablauf)

`automation/worker.py`: Läuft eine IServ-Session ab (Idle zwischen Schülern
oder mitten im Vorgang), erkennt der Worker den Login-Redirect und meldet den
Browser-Context neu an, dann wird die Kartei neu geladen.

- Erkennung `_on_login_page()`: URL enthält `iserv/login`/`iserv/auth` **oder**
  `input[name="_username"]` im DOM.
- `_goto_authed()`: Navigation mit automatischem Re-Login + Wiederholung.
- `load_card()` nutzt das für App-Root und `#/counter`; `submit_barcode()`
  erholt sich einmalig, falls das Barcode-Feld verschwunden ist.
- Re-Login nutzt die bestehende `WorkerPool._login()` (gleicher read-only
  Admin-Account), kein neuer Login-Pfad.

**Deterministischer Test:** `automation/recovery_test.py` erzwingt den Ablauf
via `context.clear_cookies()` und prüft, dass `open_student()` trotzdem
durchläuft und die Kartei nutzbar ist (`staged`). **Ergebnis: bestanden**
(Log zeigt „Session abgelaufen … Re-Login").

## 3. Paralleltest + Pool-Härtung (Nachtrag 2026-06-15)

- **2-Helfer-Paralleltest** (`automation/e2e_parallel.py`): zwei Helfer, zwei
  Schüler gleichzeitig „Aktiv". Beide Schülerkarteien laden parallel über
  getrennte Browser-Contexts, beide Scanner stagen unabhängig einen Barcode —
  keine Vermischung. **Bestanden** (verifizierte O2 jetzt auch im vollen
  UI-Flow, nicht nur im Spike).
- **Pool-Härtung** (`WorkerPool.start()`): fehlgeschlagene Worker-Logins werden
  einmal nachgezogen (Retry), gescheiterte Contexts sauber geschlossen
  (kein Leak). Faktorisiert in `_make_logged_in_context()`.
